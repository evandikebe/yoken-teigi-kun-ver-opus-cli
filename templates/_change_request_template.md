# 変更要求台帳 (change_requests.md)

> 設計確定後・実装中に発生した仕様変更を記録する台帳。spec-change-manager が管理する。
> **記録のない変更は存在しない扱い**。どんな小さな変更もここに残すこと。

最終更新: <YYYY-MM-DD>

---

## ステータス定義

- `open` — 起票済み・影響分析中
- `approved` — ユーザー承認済み・更新作業中
- `done` — ドキュメント更新 + 再レビュー完了
- `deferred` — 保留（バックログ）
- `rejected` — 見送り（理由を残す）

---

## CR 一覧（サマリ）

| CR | 日付 | 一行サマリ | 種別 | 緊急度 | 状態 |
|---|---|---|---|---|---|
| CR-001 | YYYY-MM-DD | <例: 注文キャンセル機能の追加> | 追加 / 変更 / 削除 | 今すぐ / 次MS / バックログ | open |

---

## CR 詳細（追記式）

### CR-001: <変更の一行サマリ>

- **起票日時**: YYYY-MM-DD HH:MM
- **要求者**: <ユーザー名>
- **種別**: 追加 / 変更 / 削除
- **緊急度**: (1) 今すぐ / (2) 次のマイルストーンまで / (3) バックログ
- **背景（なぜ変えるか）**: ...
- **変更内容**: ...

#### 影響分析

- **関連仕様 ID**: F-XXX, SC-XXX, EP-XXX（新規追加の場合は予約 ID）
- **影響フェーズ**: Phase 1 / 2 / 3 / 4 / 5
- **影響ドキュメント**:
  - docs/01_requirements/03_機能要件.md（§...）
  - docs/03_detailed_design/01_API仕様.md（EP-XXX）
- **影響実装ファイル**（実装中の場合・@spec 逆引き結果）:
  - src/...（チケット T-XXX / status: done → 要改修）
- **コスト概算への影響**: なし / あり（Phase 5 更新要）

#### 承認

- **承認日時**: YYYY-MM-DD HH:MM
- **承認内容**: (1) 全量で進める / (2) スコープ縮小（内容: ...）

#### 実施記録

- 更新したファイル: ...
- spec-critic 再レビュー: PASS / PASS_WITH_CONDITIONS（docs/_state/phase_reviews/...）
- 実装への申し送り: 差分チケット起票依頼済み / 該当なし
- コミット: `docs(change): CR-001 ...`
- **状態**: done
