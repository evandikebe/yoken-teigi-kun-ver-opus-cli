# 設計エージェント 共通ルール (SPEC_RULES)

> このファイルは **設計フェーズの全エージェント**（spec-orchestrator / solution-architect / requirements-analyst / basic-designer / ui-mock-designer / detailed-designer / cost-estimator / spec-handoff-writer / spec-critic / spec-change-manager）が共有する不変ルール集です。
> ※ spec-handoff-writer と spec-critic はユーザーへの質問をしないエージェントのため、Q-1 は適用外（Q-3〜Q-5 は適用）。
> 各エージェント定義の「絶対に守るルール」から参照されます。エージェント固有のルールは各定義ファイル側に書きます。

## 参照方法

- プラグインとして導入している場合: `${CLAUDE_PLUGIN_ROOT}/references/SPEC_RULES.md`
- 手動コピーで導入している場合: `.claude/references/SPEC_RULES.md`（install.md の手順で配置）

---

## Q-1: 対話方式（最重要）

1. **質問はチャットに番号付きの箇条書きで書き、ユーザーの返信を待つ**。**`AskUserQuestion` ツールは使わない**。
2. 各質問には **選択肢（例: (1)/(2)/(3)）** と、ジュニア・非エンジニアにも分かる **1行補足** を添える。
3. **1回の質問は最大4問まで**。ユーザーを疲れさせない。
4. **専門用語は1行で噛み砕いて添える**。例：「冪等性（同じ操作を何度やっても結果が同じになる性質）」「RPO（障害時にどこまで遡ってデータを失ってよいか）」
5. **「わからない」「未定」も正当な回答** として受け入れ、デフォルト値を提示して後で詰める。即決を強要しない。
6. **回答で新しい不明点が増えたら、その分も追加で質問する**。回答内容に矛盾がある場合も再確認する。
7. **情報が揃いきるまで質問を続ける**。ユーザーが明示的に「もう十分」と言った場合のみ、未確定部分を明記したうえで完了とする。

### 質問設計の例

- ❌ 「SLA は何 % ですか？」
- ✅ 「サービスが止まっても許される時間はどれくらいですか？」
  - (1) 1日数時間止まっても困らない（99%）
  - (2) 月に数時間までなら許容（99.9%）
  - (3) ほぼ止められない（99.99% 以上）

## Q-2: インライン実行の原則（オーケストレーター向け）

**ユーザーへの対話的な質問は、メインセッションで動作している時（インライン実行時）にしか成立しない**。`Agent` ツールで起動したサブエージェントは、出力がユーザーに届かず返信も受け取れない。

- ユーザーへの質問が必要なフェーズは、オーケストレーターが **該当エージェントの定義を Read してインラインで直接実行** する。
- 質問が不要な純粋な生成作業のみ `Agent` ツールで委譲してよい。

---

## Q-3: 記録と状態管理

1. **回答は逐次ファイルに保存する**。`docs/_state/answers.md` に質問本文・選択肢・回答・タイムスタンプ（YYYY-MM-DD HH:MM）を追記する。
2. **未解決事項は `docs/_state/open_questions.md` にチケット化** して残す。解消するか「実装中に決める」と明示するまでクローズしない。
3. **フェーズの進捗・承認状態は `docs/_state/phase_status.md` に反映** する。

---

## Q-4: 成果物の置き場所

1. **設計ドキュメントは `docs/` フォルダ1つに集約** する。散らばらせない。
2. **コードは `docs/` に置かない**（コードは実装フェーズで `src/` 配下へ）。設計ドキュメント内でコードを示す場合は Markdown のコードブロックとして書く。
3. テンプレートはプラグインの `templates/` 配下にある（`${CLAUDE_PLUGIN_ROOT}/templates/`）。新規作成時はテンプレートをコピーして使う。

---

## Q-5: 品質の最低ライン

1. **機能要件と非機能要件の両方を必ず確定させる**。非機能要件は忘れられがちなので意識的に深掘りする。
2. **承認ゲートを飛ばさない**。Phase 0 の承認が取れるまで Phase 1 以降に進まない。
3. **画面デザインを勝手に決めない**。トーン・配色・レイアウトは必ずユーザーに確認する。
4. **技術選定・設計判断には理由を書く**。「なんとなく」は禁止。

---

## Q-6: 再開（レジューム）の原則

セッションは途中で切れる前提で動く。**記憶ではなく状態ファイルが唯一の真実**。

1. **起動時に必ず既存状態を確認する**。`docs/_state/phase_status.md`（フェーズ進捗・critic 判定）、`docs/_state/answers.md`（Q&A履歴）、`docs/_state/open_questions.md`、`docs/_state/change_requests.md`（あれば）を Read してから動き出す。
2. **既存プロジェクトなら「再開宣言」をしてから続行する**。例:「Phase 2 の途中からの再開なのだ。画面一覧まで完了・critic 未実施。画面遷移図の作成から再開する」— 現在地・完了済み・次の一手の3点をユーザーに提示し、認識がズレていれば番号付き質問で確認する。
3. **完了済みフェーズをやり直さない**。critic PASS 済みのフェーズは原則そのまま（変更したい場合は spec-change-manager の変更管理フローへ）。
4. **作業の区切りごとに状態ファイルを更新する**。次のセッションの自分が読むつもりで、「どこまで終わったか・次に何をするか」を phase_status.md に残す。
